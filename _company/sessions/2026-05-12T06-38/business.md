# 💰 Business — Data Integrity Protocol(DIP)이 실제 데이터 입력 단계부터 강제되도록 시스템 요구사항을 최종 확정하고, 이에 대한 운영 절차 문서를 작성하라.

💰 Business: 작업 시작합니다.

CEO의 지시에 따라 'Data Integrity Protocol (DIP)'을 시스템 전반에 걸쳐 강제하기 위한 최종 요구사항과 운영 절차 문서를 작성했습니다. 이 문서는 단순히 기술적인 검증 단계를 넘어, **데이터를 사용하는 모든 의사결정 과정**에서 신뢰도를 최우선으로 하도록 하는 회사 운영의 핵심 원칙이 됩니다.

이 문서는 향후 Developer 에이전트가 API 스키마에 반영해야 할 *기술 요구사항*과, Designer/Analyst 에이전트가 실제 보고서를 만들 때 지켜야 할 *운영 절차*를 모두 포함합니다.

---
### 📂 DIP 운영 매뉴얼: MEA SalesCompass 데이터 무결성 프로토콜 (v1.0)

**문서 목적:** 모든 내부 및 외부 보고서, KPI 산출물의 근간이 되는 원본 데이터가 입력 단계부터 검증되어 신뢰성을 확보하는 표준 절차를 확립합니다.
**적용 범위:** 데이터 수집(Collection) $\to$ 전처리/분석(Processing) $\to$ 시각화/보고(Output)의 전 과정.

#### 1. DIP 핵심 원칙 (Core Principles)

| 원칙 | 내용 | 비즈니스 영향 (Why?) |
| :--- | :--- | :--- |
| **투명성 (Transparency)** | 모든 KPI는 반드시 데이터 출처(Source Link/DB ID)를 명시해야 한다. (Audit Trail 필수) | '이 숫자가 어디서 왔는지'에 대한 신뢰도를 1순위로 확보한다. |
| **재현 가능성 (Reproducibility)** | 특정 분석 결과를 도출하는 모든 계산식과 전처리 로직은 문서화되고, 코드로 버전 관리되어야 한다. | 다른 사람이 같은 데이터를 가지고도 동일한 결론에 도달할 수 있음을 증명한다. |
| **불확실성 명시 (Uncertainty)** | 데이터의 공백(Gap)이나 예측 모델 기반 값에는 반드시 신뢰도 점수(Confidence Score)와 함께 '주의' 플래그를 붙인다. | 불완전한 정보로 인한 오판을 방지하고, 의사결정권자에게 위험 인식을 제공한다. |

#### 2. 시스템 요구사항 (System Requirements for Developer)

이 섹션은 개발팀(Developer Agent)이 데이터 파이프라인 및 API 스키마에 반드시 반영해야 하는 **기술적 강제 로직**입니다.

| 요구 사항 | 설명 | 구현 방식 |
| :--- | :--- | :--- |
| **[R1] 필수 필드 검증 (Mandatory Check)** | 핵심 KPI 계산에 사용되는 모든 입력 데이터 포인트는 Null 또는 Empty String이 아니어야 한다. | API 게이트웨이 레벨에서 `Schema Validation`을 강제하고, 실패 시 에러 코드를 반환한다. |
| **[R2] 범위 검증 (Range/Type Check)** | 데이터가 사전에 정의된 비즈니스 허용 범위를 벗어나거나(예: 매출액 < 0), 데이터 타입이 일치하지 않을 경우를 탐지해야 한다. | `If-Else` 로직을 적용하여 이상치를 플래그 처리하고, 자동 수정 대신 경고 로그(Warning Log)를 생성한다. |
| **[R3] 출처 메타데이터 삽입 (Provenance Tagging)** | 데이터가 시스템에 입력되는 순간, 원본 데이터의 ID, 수집 날짜, 담당자, 그리고 최초 로딩된 API 엔드포인트 정보를 자동으로 태그로 삽입해야 한다. | 모든 DB 레코드 또는 Data Frame 객체에 `_source_meta` 필드를 강제한다. (변경 불가능) |
| **[R4] 데이터 버전 관리 (Version Control)** | 원본 데이터가 수정되거나 새로운 로직이 추가될 경우, 이전 상태를 보존하는 스냅샷 메커니즘을 구현해야 한다. | ETL(Extract-Transform-Load) 파이프라인에 `Snapshot` 단계를 의무화하고, 각 버전별로 Index를 부여한다. |

#### 3. 운영 절차 (Operational Procedure for Analyst/Designer)

이 섹션은 데이터를 분석하고 보고서를 제작하는 모든 인력(실제 사용자 및 에이전트)이 따라야 할 **워크플로우 표준**입니다.

1.  **[Step 1: 데이터 요청 시]**: 필요한 KPI를 정의할 때, 반드시 해당 KPI에 대한 **데이터 출처 리스트(Source List)**와 예상되는 **최소/최대 허용 범위**를 먼저 제출해야 한다. (지원을 받는 사람이 아닌, 데이터를 정의하는 쪽에서 주도)
2.  **[Step 2: 데이터 검증 및 로딩]**: 시스템에 접근하여 데이터를 로드할 때, DIP 대시보드를 통해 **데이터 신뢰도 점수(DIP Score)**를 확인한다. (점수가 낮으면 [3단계]로 이동).
3.  **[Step 3: 결함 발생 시 처리]**: DIP Score가 임계치 이하일 경우, 분석을 중단하고 다음 세 가지 액션 중 하나를 수행해야 한다.
    *   **(A) 데이터 Gap 채우기:** 원본 소스에서 누락된 데이터를 찾고, [R3] 프로토콜에 따라 수동으로 출처 메타데이터를 삽입한다.
    *   **(B) 가정(Assumption) 명시:** 공백을 임의로 채울 수밖에 없다면, 해당 지표 옆에 **"가정치 (Assumption: 202X년 X월 Y일 기준 추정)"**라는 고지 문구를 반드시 넣는다.
    *   **(C) 보고서 제외/보류:** 데이터 신뢰도 확보가 불가능한 경우, 분석에서 해당 KPI를 즉시 제거하고 '데이터 부족으로 보류' 처리한다.
4.  **[Step 4: 최종 검토 및 승인]**: 보고서 배포 전, **최종 DIP 체크리스트**를 통해 모든 그래프와 수치가 출처 메타데이터와 논리 흐름을 갖추었는지 크로스체크하는 절차를 의무화한다.
