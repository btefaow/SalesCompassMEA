| **5. 행동 촉구 (Call to Action)** | 결단, 긴급함 ("지금 당장 시작해야 해.") | *[최종 경고]* 이 시스템 도입 지연은 곧 '미래 손실'로 이어집니다. 지금이 바로 리스크를 회피할 골든타임입니다. | ROI 요약 (숫자 중심) 및 단계별 로드맵 제시. (예: 1단계 실행 시, 최소 $X\%$의 $\text{Loss}_{Gap}$ 해소). | **톤:** 긴급하고 명확함(Urgent & Clear). 가장 강력한 액센트 컬러와 숫자를 조합하여 '결정'을 강요하는 듯한 압박감을 주되, 신뢰를 잃지 않도록 합니다. |

1. **언어 사용:** "발견했습니다" $\rightarrow$ "**지금 놓치고 있는 리스크입니다.**"로 변경. (정보 전달 $\to$ 위협 감지).

## 📐 파트 A: 전체 레이아웃 규격 (Global Grid & Typography)

### 1. 그리드 시스템 (Grid System)
*   **기준 크기:** A4 Portrait (210mm x 297mm).
*   **컬럼 구조:** 3-Column Grid System을 기본으로 사용합니다.
    *   좌측/우측 여백(Gutter): 최소 25mm 확보 (시각적 숨 쉴 공간 제공).
    *   메인 콘텐츠 영역: 전체 너비의 약 80%를 차지하며, 가로 세 개의 균등한 컬럼으로 나뉩니다.
*   **헤더/푸터:** 페이지 번호는 우측 하단에 작고 간결하게 배치합니다. 로고와 섹션 제목은 좌측 상단에 반복적으로 노출하여 통일성을 강조합니다.

### 2. 타이포그래피 (Typography)
| 요소 | 폰트명 | 용도 및 스타일 | 크기(pt) | 색상 코드 |
| :--- | :--- | :--- | :--- | :--- |
| **제목 (H1)** | Noto Sans KR Bold | 보고서의 핵심 주제. 시선을 압도하며 중요성을 부여. | 36~48 pt | Deep Sapphire Blue (#0A2342) |
| **소제목 (H2)** | Noto Sans KR SemiBold | 섹션 구분 및 논리적 구조 제시. | 24~30 pt | Alert Orange (#D9534F) |
| **본문 (Body Text)** | Noto Sans KR Regular | 데이터 분석 결과, 설명 문구 등 일반 서술. 가독성이 최우선. | 10~12 pt | #333333 (진한 회색) |
| **캡션/각주** | Noto Sans KR Light | 그래프 하단 주석 및 출처 표기. | 8~9 pt | #777777 (옅은 회색) |

### 3. 컬러 팔레트 활용 규칙 (Color Usage Map)
*   **Deep Sapphire Blue (#0A2342):** 신뢰, 안정성, 전문성을 상징하는 주 색상. **(헤더, 핵심 결론 강조, 구조적 요소)**에 사용합니다.
*   **Alert Orange (#D9534F):** 문제 제기, 위험 경고, 즉각적인 행동 촉구(CTA)를 나타내는 보조색. **(문제 지표 하이라이트, Call-to-Action 버튼/박스)**에 제한적으로 사용합니다.
*   **Neutral Gray (#777777):** 부가 정보, 출처, 구분선 등 시선을 분산시키지 않는 요소에만 사용합니다.

## 🖼️ 파트 B: 핵심 템플릿 컴포넌트 (Wireframes & Design Specs)

보고서의 논리 흐름(Flow Map V4.0)을 따라 다음 3가지 필수 페이지 유형을 표준화합니다.

### 1. [Page Type A] 문제 정의 및 충격 인식 페이지 (The Crisis Page)
*   **목적:** 청중에게 '현재 상황의 위험성'을 즉각적으로 인지시키고, 공포심(Fear of Loss)을 자극하는 것이 목표입니다.
*   **레이아웃 (Wireframe):**
    1.  **H1 제목:** 페이지 상단에 배치 (Deep Sapphire Blue). 충격적인 질문형 제목 사용 예: "당신의 비즈니스는 이 Gap 때문에 멈출 수 있습니다."
    2.  **핵심 시각화(Hero Chart):** 중앙의 가장 큰 영역을 차지합니다. **'Gap 크기'를 한눈에 보여주는 단일 차트(예: Funnel Diagram 또는 Dual Bar Chart)**가 필수입니다. (좌우 대비 구조 활용).
    3.  **세부 분석 박스 (Mini Insights):** 3개의 작은 컬럼으로 구성되며, 각 컬럼에는 '문제 지표'와 그에 대한 간결한 설명(Body Text)이 배치됩니다. 경고의 의미를 담아 **Alert Orange**로 테두리를 처리합니다.
    4.  **액션 유도:** 하단에 "➡️ 다음 페이지에서 해결책을 제시하겠습니다." 등의 흐름 안내 문구 삽입.

### 2. [Page Type B] 데이터 기반 솔루션 및 분석 페이지 (The Solution Proof Page)
*   **목적:** '시스템의 부재'가 근본적인 문제였음을 증명하고, 우리의 솔루션을 가장 논리적으로 위치시키는 페이지입니다.
*   **레이아웃 (Wireframe):**
    1.  **H2 제목:** 섹션의 논리를 담는 구체적인 메시지 제시 (Deep Sapphire Blue). 예: "단순 자본 투입을 넘어선 '구조적 리스크 관리'가 필요합니다."
    2.  **모델 아키텍처 다이어그램 (Core Component):** 페이지 중앙에 배치되는 가장 중요한 요소입니다. 우리의 솔루션이 어떻게 작동하는지 **3단계 Flow Diagram** 형태로 시각화합니다. (A $\rightarrow$ B $\rightarrow$ C)
        *   *디자인 지침:* 각 단계는 명확한 아이콘과 함께 사각형 박스로 구분하고, 화살표(Arrow)를 통해 인과관계를 강조해야 합니다.
    3.  **근거 데이터 시각화:** 2~3개의 그래프가 배치됩니다. 단순 막대/파이 차트보다 **'Before & After' 비교가 가능한 라인 차트 또는 트렌드 플롯**을 우선 사용합니다. (데이터의 '변화'에 초점을 맞춥니다.)

### 3. [Page Type C] 최종 CTA 및 구매 결정 페이지 (The Monetization Point Page)
*   **목적:** 고객이 스스로 필요성을 느끼고, **가장 쉬운 옵션부터 가장 완벽한 통합 솔루션까지** 선택하도록 유도하는 페이지입니다.
*   **레이아웃 (Wireframe):**
    1.  **핵심 메시지 재강조:** 상단에 H2 제목과 함께 최종적인 'Loss Avoidance'의 가치를 감성적으로 요약합니다. (Deep Sapphire Blue)
    2.  **옵션 비교 테이블 (The Choice Matrix):** 페이지의 60%를 차지하는 표 형태입니다. 최소한 다음 세 가지 옵션을 명확하게 대비시켜야 합니다.
        *   **Option 1 (Entry Level):** 낮은 가격, 제한적 기능. (가장 적은 노력으로 시작할 수 있다는 안도감 제공)
        *   **Option 2 (Recommended/Best Value):** 핵심 솔루션 패키지. **Alert Orange로 테두리를 강조**하여 시선을 유도합니다.
        *   **Option 3 (Premium/Full Integration):** 최고 가치, 모든 리스크 제거. (최종적인 '만족감'을 제공)
    3.  **Call to Action (CTA):** 페이지 최하단에 전면적으로 배치되는 박스입니다. 깊고 진한 배경색과 함께 "지금 문의하세요" 같은 명확하고 쉬운 행동 지침이 포함되어야 합니다.

<create_file path="Design/Tech_Specs/VisualizationComponent_TSpec_v3.0.md"># 💻 VisualizationComponent 기술 사양서 (Technical Specification V3.0)

## 📄 1. 개요 및 역할
*   **목적:** MEA SalesCompass 보고서 및 자료에 사용될 핵심 시각화 데이터 컴포넌트의 통합 가이드라인을 정의합니다. 모든 Mockup 개발팀(디자이너, 프론트엔드)이 이 사양서를 최종 개발 기준으로 삼아야 합니다.
*   **기능 범위:** Loss $\rightarrow$ FLI $\rightarrow$ ROI의 재무적 흐름과 그에 따른 '데이터 신뢰도 지표'를 시각화합니다. (비용 절감 대신 손실 회피 관점 강조).

## 💾 2. 데이터 스키마 및 입력값 (Input Schema)
컴포넌트가 받아야 하는 필수 JSON 데이터 구조입니다. 모든 값은 숫자형(Number)이어야 하며, 백엔드 파이프라인에서 반드시 검증되어 전달됩니다.

```json
{
  "date_range": "YYYY-MM-DD to YYYY-MM-DD", // 보고서 기간
  "metrics": {
    "loss_gap": 1234567.89,      // A. 초기 잠재적 손실 (Loss_{Gap}) - 필수
    "fli_value": 890123.45,       // B. 개선 후 예상 수익 증대분 (FLI) - 필수
    "roi_potential": 234567.89,   // C. 최종 ROI 잠재력 (ROI) - 필수
    "trust_score": 0.85           // 데이터 신뢰도 점수 (0.0 ~ 1.0) - 필수
  },
  "data_points": [
    { "label": "Metric A", "value": 1234567, "type": "LossGap" },
    { "label": "Metric B", "value": 890123, "type": "FLI" },
    // ... 최대 3개까지 포인트가 들어갈 수 있습니다.
  ]
}
```

## ✨ 3. 컴포넌트 Props 및 커스터마이징 (Frontend Integration)
프론트엔드 개발 시 사용 가능한 주요 속성(Props) 목록입니다.

| Prop Name | Type | Required | Description | Default Value |
| :--- | :--- | :--- | :--- | :--- |
| `data` | Object | Yes | 위에 정의된 Input Schema를 준수해야 합니다. | N/A |
| `title` | String | No | 시각화 섹션의 제목 (예: "잠재적 손실 회피 규모") | "핵심 재무 위험 분석" |
| `colorScheme` | String | No | 사용될 색상 팔레트 (`'primary'`, `'warning'`, `'success'`) | `'primary'` |

## ⚠️ 4. 예외 처리 및 데이터 무결성 로직 (Error Handling & Validation)
이 부분이 가장 중요합니다. 백엔드에서 다음 예외 상황 발생 시, 컴포넌트는 이 규칙에 따라 작동해야 합니다.

1.  **API 연결 실패:** 데이터를 가져올 수 없을 경우, `metrics` 객체 전체를 비활성화하고 대신 "데이터 로딩 불가: [YYYY-MM-DD] 기준 데이터 확인 필요"라는 텍스트만 표시합니다.
2.  **필수 데이터 누락 (Loss Gap):** `loss_gap` 값이 없는 경우, 컴포넌트는 강제로 '0'으로 처리하지 않고, 대신 해당 섹션의 모든 시각 자료를 **비활성화(Disabled)**하고 경고 메시지를 출력해야 합니다. *(`$Loss_{Gap}$는 논리적 기반이므로 절대 0으로 처리 불가)*.*
3.  **신뢰도 점수 (Trust Score):** `trust_score`가 임계치($T_{crit}=0.7$) 이하일 경우, 시각화 그래프의 모든 데이터 포인트에 **경고(Warning)** 아이콘을 표시하고, 해당 섹션 하단에 "데이터 신뢰도가 낮습니다. 추가 검증이 필요합니다."라는 문구를 삽입해야 합니다.

## 🧪 5. 테스트 케이스 (Test Cases)
*   **Happy Path:** 모든 데이터가 정상적으로 주입되고 시각화됨. (TSpec V1.0 기준)
*   **Failure Case (Loss Gap):** `loss_gap`이 누락되었을 때, 컴포넌트 전체가 비활성화되는지 테스트합니다.
*   **Warning Case:** `trust_score`가 0.65로 설정되어 경고 문구가 올바르게 표시되는지 테스트합니다.
